---
title: "Using Taskiq with FastAPI"
description: "Connect taskiq's async task queue to FastAPI so tasks can resolve FastAPI dependencies, share app state, and run alongside your ASGI server."
---

Taskiq ships a first-party integration library, **taskiq-fastapi**, that lets you reuse FastAPI's dependency injection system inside background tasks. This page explains how to install and configure the integration, how to annotate dependencies so they work in both request handlers and tasks, and how to write tests.

## Installation

<Steps>
  <Step title="Install the integration library">
    ```bash
    pip install taskiq-fastapi
    ```
  </Step>

  <Step title="Initialize the integration in your broker file">
    Call `taskiq_fastapi.init` in the same file where you define your broker. The second argument is a dotted import path to your FastAPI application (the same string format used by Uvicorn).

    ```python
    from taskiq import ZeroMQBroker
    import taskiq_fastapi

    broker = ZeroMQBroker()

    taskiq_fastapi.init(broker, "my_package.application:app")
    ```
  </Step>

  <Step title="Start and stop the broker in your app's lifespan">
    Register broker startup and shutdown inside your application's lifespan context manager. Guard each call with `broker.is_worker_process` so the web server does not try to act as a worker.

    <Tabs>
      <Tab title="Lifespan (recommended)">
        ```python
        from contextlib import asynccontextmanager
        from fastapi import FastAPI
        from your_project.taskiq import broker


        @asynccontextmanager
        async def lifespan(app: FastAPI):
            # Startup
            if not broker.is_worker_process:
                await broker.startup()
            yield
            # Shutdown
            if not broker.is_worker_process:
                await broker.shutdown()


        app = FastAPI(lifespan=lifespan)
        ```
      </Tab>
      <Tab title="on_event (deprecated)">
        ```python
        from fastapi import FastAPI
        from your_project.taskiq import broker

        app = FastAPI()


        @app.on_event("startup")
        async def app_startup():
            if not broker.is_worker_process:
                await broker.startup()


        @app.on_event("shutdown")
        async def app_shutdown():
            if not broker.is_worker_process:
                await broker.shutdown()
        ```
      </Tab>
    </Tabs>
  </Step>
</Steps>

## The `is_worker_process` flag

Every `AsyncBroker` instance exposes an `is_worker_process` attribute (default `False`). Taskiq's worker process sets this flag to `True` before calling `broker.startup()`. Use it to conditionally run broker lifecycle calls so that only one side — worker or web server — owns the connection.

```python
if not broker.is_worker_process:
    await broker.startup()
```

## Using FastAPI dependencies in tasks

There are two rules to follow when you want a task to resolve dependencies that were written for FastAPI request handlers.

### Rule 1 — mark `Request` and `HTTPConnection` parameters with `TaskiqDepends`

Many FastAPI dependencies accept a `fastapi.Request` parameter to reach `request.app.state`. Taskiq cannot inject these automatically unless you mark them explicitly.

<Tabs>
  <Tab title="Annotated (Python 3.10+)">
    ```python
    from typing import Annotated
    from fastapi import Request
    from taskiq import TaskiqDepends


    async def get_redis_pool(request: Annotated[Request, TaskiqDepends()]):
        return request.app.state.redis_pool
    ```
  </Tab>
  <Tab title="Default values">
    ```python
    from fastapi import Request
    from taskiq import TaskiqDepends


    async def get_redis_pool(request: Request = TaskiqDepends()):
        return request.app.state.redis_pool
    ```
  </Tab>
</Tabs>

### Rule 2 — use only `TaskiqDepends` in tasks

Inside a task function, declare every injected parameter with `TaskiqDepends` as its default value (or `Annotated` equivalent). Standard `fastapi.Depends` is not resolved outside a request context.

```python
from taskiq import TaskiqDepends
from your_project.deps import get_redis_pool


@broker.task
async def send_notification(user_id: int, redis=TaskiqDepends(get_redis_pool)):
    pool = await redis
    await pool.publish(f"notify:{user_id}", "hello")
```

<Note>
  The `Request` object injected into your task is a **mocked request**, not the original HTTP request that triggered the task. It exists solely so that dependencies written against `Request` can be resolved by taskiq's dependency container.
</Note>

## Testing

When using `InMemoryBroker` in tests, the broker does not run as a worker process, so you must populate the dependency context manually. The `taskiq_fastapi.populate_dependency_context` helper does this for you.

Define a fixture that creates your FastAPI app, then a second `autouse` fixture that populates the context before each test and clears it after:

```python
import pytest
import taskiq_fastapi
from fastapi import FastAPI
from your_project.app import get_app
from your_project.taskiq import broker


@pytest.fixture
def fastapi_app() -> FastAPI:
    return get_app()


@pytest.fixture(autouse=True)
def init_taskiq_deps(fastapi_app: FastAPI):
    # Populate the dependency context so taskiq can resolve
    # FastAPI dependencies inside InMemoryBroker tasks.
    taskiq_fastapi.populate_dependency_context(broker, fastapi_app)

    yield

    broker.custom_dependency_context = {}
```

<Tip>
  Mark the second fixture with `autouse=True` so the dependency context is always fresh for every test without requiring explicit fixture references.
</Tip>
