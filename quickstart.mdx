---
title: "Quickstart: Send Your First taskiq Task in Minutes"
description: "Install taskiq, define a task with the @broker.task decorator, send it with .kiq(), and read the result — then swap in a production-ready RabbitMQ broker."
---

This page walks you from a working local example using `InMemoryBroker` all the way to a distributed setup with RabbitMQ and a Redis result backend. Every code snippet is a real, runnable file — no placeholders.

<Steps>

<Step title="Install taskiq">

Install the core package from PyPI:

<Tabs>
  <Tab title="pip">
    ```bash
    pip install taskiq
    ```
  </Tab>
  <Tab title="uv">
    ```bash
    uv add taskiq
    ```
  </Tab>
  <Tab title="poetry">
    ```bash
    poetry add taskiq
    ```
  </Tab>
</Tabs>

The core package includes `InMemoryBroker` for local development. You don't need any external services to run the next step.

</Step>

<Step title="Write and run your first task">

Create a file called `broker.py` with the following contents:

```python
# broker.py
import asyncio

from taskiq import InMemoryBroker

broker = InMemoryBroker()


@broker.task
async def add_one(value: int) -> int:
    return value + 1


async def main() -> None:
    # Never forget to call startup in the beginning.
    await broker.startup()
    # Send the task to the broker.
    task = await add_one.kiq(1)
    # Wait for the result.
    result = await task.wait_result(timeout=2)
    print(f"Task execution took: {result.execution_time} seconds.")
    if not result.is_err:
        print(f"Returned value: {result.return_value}")
    else:
        print("Error found while executing task.")
    await broker.shutdown()


if __name__ == "__main__":
    asyncio.run(main())
```

Run it:

```bash
python broker.py
```

Expected output:

```
Task execution took: 7.3909759521484375e-06 seconds.
Returned value: 2
```

<Note>
`InMemoryBroker` executes tasks in the same process as the caller. It is useful for development and testing but does not send any data over the network. You cannot use it to run tasks in a separate worker process.
</Note>

**What just happened?**

- `@broker.task` registers `add_one` as a task and returns an `AsyncTaskiqDecoratedTask` object.
- `.kiq(1)` assembles a message with the argument `1` and sends it. It returns an `AsyncTaskiqTask` with a `task_id`.
- `.wait_result(timeout=2)` polls the result backend until the result is ready, then returns a `TaskiqResult`.
- `result.is_err` is `False` when the task completed without raising an exception. `result.return_value` holds the return value.

<Warning>
Always call `await broker.startup()` before sending any tasks. Skipping this can cause undefined behavior, especially with brokers that need to establish a network connection.
</Warning>

</Step>

<Step title="Switch to a distributed broker">

`InMemoryBroker` is not suitable for production. The following example replaces it with `AioPikaBroker` (RabbitMQ) and adds a Redis result backend so results survive across processes.

Install the additional packages:

```bash
pip install taskiq-aio-pika taskiq-redis
```

Start RabbitMQ and Redis with Docker:

<Tabs>
  <Tab title="Linux / macOS">
    ```bash
    docker run --rm -d \
        -p "5672:5672" \
        -p "15672:15672" \
        --env "RABBITMQ_DEFAULT_USER=guest" \
        --env "RABBITMQ_DEFAULT_PASS=guest" \
        --env "RABBITMQ_DEFAULT_VHOST=/" \
        rabbitmq:3.8.27-management-alpine

    docker run --rm -d \
        -p "6379:6379" \
        redis
    ```
  </Tab>
  <Tab title="Windows">
    ```powershell
    docker run --rm -d ^
        -p "5672:5672" ^
        -p "15672:15672" ^
        --env "RABBITMQ_DEFAULT_USER=guest" ^
        --env "RABBITMQ_DEFAULT_PASS=guest" ^
        --env "RABBITMQ_DEFAULT_VHOST=/" ^
        rabbitmq:3.8.27-management-alpine

    docker run --rm -d ^
        -p "6379:6379" ^
        redis
    ```
  </Tab>
</Tabs>

Update `broker.py` to use the distributed broker and result backend:

```python
# broker.py
import asyncio

from taskiq_aio_pika import AioPikaBroker
from taskiq_redis import RedisAsyncResultBackend

broker = AioPikaBroker(
    "amqp://guest:guest@localhost:5672",
).with_result_backend(RedisAsyncResultBackend("redis://localhost"))


@broker.task
async def add_one(value: int) -> int:
    return value + 1


async def main() -> None:
    await broker.startup()
    # Send the task to the broker.
    task = await add_one.kiq(1)
    # Wait for the result.
    result = await task.wait_result(timeout=2)
    print(f"Task execution took: {result.execution_time} seconds.")
    if not result.is_err:
        print(f"Returned value: {result.return_value}")
    else:
        print("Error found while executing task.")
    await broker.shutdown()


if __name__ == "__main__":
    asyncio.run(main())
```

<Tip>
For production, `taskiq-aio-pika` (RabbitMQ) or `taskiq-nats` are recommended as the broker, and `taskiq-redis` as the result backend. Check PyPI for the full list of available broker packages by searching `taskiq`.
</Tip>

</Step>

<Step title="Start the worker process">

With a distributed broker, tasks are executed by a separate worker process. Start one with the `taskiq worker` CLI command, pointing it at the module and variable name of your broker:

```bash
taskiq worker broker:broker
```

The format is `module_path:variable_name`. Here `broker` is both the filename (`broker.py`) and the variable name inside it.

If your tasks are spread across multiple modules, you can either list them explicitly or use `--fs-discover` to auto-import all `tasks.py` files recursively:

```bash
# Explicit imports
taskiq worker myproject.broker:broker myproject.tasks myproject.jobs

# Auto-discover all tasks.py files
taskiq worker myproject.broker:broker --fs-discover
```

For hot-reloading during development, install the `reload` extra and pass `--reload`:

```bash
pip install "taskiq[reload]"
taskiq worker broker:broker --reload
```

</Step>

<Step title="Send a task and check the result">

With the worker running in one terminal, run the client script in another:

```bash
python broker.py
```

Expected output:

```
Task execution took: 1.0013580322265625e-05 seconds.
Returned value: 2
```

The `main()` function sends the task via RabbitMQ, the worker picks it up and executes it, stores the result in Redis, and then `.wait_result()` fetches it back.

</Step>

</Steps>

## Setting task timeouts

You can restrict how long a task is allowed to run by adding a `timeout` label. This works both on the decorator and at call time:

<Tabs>
  <Tab title="Via decorator">
    ```python
    import asyncio

    @broker.task(timeout=0.1)
    async def slow_task() -> None:
        await asyncio.sleep(2)
    ```
  </Tab>
  <Tab title="Via kicker at call time">
    ```python
    await slow_task.kicker().with_labels(timeout=0.3).kiq()
    ```
  </Tab>
</Tabs>

<Warning>
Timeouts raise `TimeoutError` for async tasks. For synchronous tasks run via `run_in_executor`, Python cannot forcibly stop execution, so the timeout will raise but the function may continue running in the background.
</Warning>
